Sundance.org UI Framework

I hit reset on all my Sketch files, refactored an entire CSS library, and created a web-based styleguide and library of front-end components.

Wearing All the Hats

This is a three-part story of a project's journey from Sketch to Github, but it's also the story of one designer who grows from journeyman front-end developer to master. Ok, fine. Front-end dev is a big stretch. My javascript has been found wanting.

I. Sketch Overhaul- Sketch 41 introduced nested symbol overrides. To take advantage of this powerful new feature, I would need to overhaul much of my Sketch files.

II. CSS Methodology- Research best practices and methologies for writing a new CSS library.

III. Living Styleguide/UI Framework- Build a web-based style guide and library of patterns, which includes design documentation and code snippets.

I. Design Debt

After doing some extensive reading, particularly a few very excellent pieces by Jon Moore and Lloyd Humphries, it became immediately clear just how much I was under-utilizing Sketch. Their application of deep-nested symbols enables designers to build nearly every element of a screen as a symbol, which would significantly reduce time spent assembling and updating designs. I started building many of the UI components from scratch and quickly realized I was staring down the barrel of a mountain of work.

Using UX Power Tools Toolbox as a starting point (and with a hefty amount of elbow grease) I recreated all of my design patterns and prototypes for Sundance.org. I also took the opportunity to make some typographic tweaks that adhere entirely to a baseline grid. I'm very excited about it.

II. Refactoring with ITCSS/BEM

CSS is only as clean and organized as its author allows it to be. In 2014 when we relaunched Sundance.org, it was my first experience writing CSS on such a large scale. My ignorance led me to making some very poor decisions early on, such as nesting selectors too heavily and not organizing our library in a way that allows for styles to be more widely reusable. My CSS bloated and too specific.

The ITCSS methodology is based on the idea of organizing CSS in an inverted triangle starting from more broad and far-reaching styles at the top of the hierarchy and building downward to increasingly specific elements. Hereâ€™s a quick breakdown.

  • 1. Settings (variables, font sizes, colors, configs)
  • 2. Tools (preprocessor mixing and functions)
  • 3. Generic (normalize, reset, box-sizing)
  • 4. Elements (elements of the DOM, no classes)
  • 5. Objects (non-cosmetic design patterns, wrappers, layouts)
  • 6. Components (cosmetic design patterns)
  • 7. Overrides (otherwise known as the shame folder)

For those interested, Harry Roberts (who created ITCSS) provides a more in-depth explanation.

I hadn't intended to employ a framework, but as luck would have it, Harry Roberts created InuitCSS. InuitCSS is a framework comprised of both ITCSS and BEM. A framework in its purest sense, InuitCSS comes only with the basic architecture of a CSS library. It contains nothing in the way of design. Music to my ears!

III. A Living Styleguide

Using Jekyll I created a static website and hosted it on Github pages. I built every pattern on Sundance.org within this new CSS framework. Now our team can access a living styleguide, complete with design documentation, live examples, and code snippets.

One of the biggest benefits of this approach is that it allowed me to write a new CSS library independent of our website's existing codebase. There was to be no confusion or crossover between me and the other developers until the entire library was rewritten. Only then would we begin integrating it with the existing codebase. This approach also circumvents that challenge of handing off design assets to developers. It also provided me more time and space to make slight adjustments to styles and experiment between my design files and code.

Epilogue

This project pushed me in some of the technical areas where I needed to grow. I now have a very firm grasp on creating and maintaining a UI framework from start to finish. I'm confident that this knowledge will be tremendously helpful on projects in the future.
