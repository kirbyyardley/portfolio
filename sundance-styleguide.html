Sundance.org UI Framework

This is a three-part story of a style guide's journey from Sketch to Github. Here I will highlight some of what I learned along the way.

Wearing All the Hats

The gap between designers and developers continues to recede. There is a great benefit to having developers involved in process of design. Conversely, designers who are contributing to the development of front-end components can be extremely helpful in creating a team's shared understanding and investment in the entirety of a project.

I. Sketch Overhaul- Sketch 41's nested symbol overrides, revamped design documents.

II. CSS Methodology- Best practices and methodologies for writing a new CSS library.

III. Abstract Refactoring- Implementing a refactor before it's implemented.

I. Sketch Overhaul (Design Debt)

After doing some extensive reading, particularly a few very excellent pieces by Jon Moore and Lloyd Humphries, it became immediately clear just how much I was under-utilizing Sketch. Their application of deep-nested symbols enables designers to build nearly every element of a screen as a symbol using overrides to fill in content. This approach to crafting design documents is extremely efficient and allows for every screen to be built almost entirely of symbols.

Rather than starting from scratch and rebuilding all of my components as symbols, I opted to use UX Power Tools Toolbox as my starting point. After some intense modifications to their entire document, I recreated all of the patterns and designs for Sundance.org. I ended up with hundreds of new symbols. Many of which are nested inside of other symbols. Looking at the screenshots below, it may seem like a mess, but there is a genius method to this madness. Definitely, check out the aforementioned blogs to learn more.

In addition to this dramatic change in formatting, I also took advantage of this opportunity to make a few tweaks, namely adding a baseline grid to our typographic ecosystem.

II. Refactoring with ITCSS/BEM

CSS is only as clean and organized as its author can dictate. My first attempt at writing CSS on such a large scale was in 2014 when we relaunched Sundance.org. I made some poor decisions along the way like organizing our CSS in a way that doesn't allow for styles to be shared more widely across the site and selectors that are way too specific.

I spent some time researching ways to refactor our CSS and had many conversations with our devs. Ultimately we narrowed things down to ITCSS as a methodology and BEM as our naming convention.

The ITCSS methodology is based on the idea of organizing CSS in an inverted triangle starting from most broad and far-reaching styles at the top of the hierarchy and building downward to increasingly specific elements. Hereâ€™s a quick breakdown:

  • 1. Settings (variables, font sizes, colors, configs)
  • 2. Tools (preprocessor mixins and functions)
  • 3. Generic (normalize, reset, box-sizing)
  • 4. Elements (HTML DOM elements, no classes)
  • 5. Objects (non-cosmetic design patterns like wrappers and layouts)
  • 6. Components (cosmetic design patterns)
  • 7. Overrides (otherwise known as the shame folder)

For those interested, Harry Roberts (the creator) provides a more in-depth explanation.

I had initially wanted to avoid the use of a framework, but as luck would have it, Harry Roberts created InuitCSS. InuitCSS is a framework comprised of both ITCSS and BEM. A framework in its purest sense, InuitCSS comes only with the basic architecture of a CSS library. It contains nothing in the way of design. Music to my ears!

This worked extremely well. InuitCSS has so much utility out of the box that you can assemble entire patterns without having to write very many additional custom classes.

III. Abstract Refactoring

One of the biggest risks when performing a refactor of this scale is that you are making sweeping changes to the codebase. As this is happening, coordinating work between the team becomes very challenging and you're now having to toil with two separate libraries of CSS at once. Our team really wanted to avoid this nightmare. I opted to write an entire abstract CSS library first.

I spun up a Jekyll instance, installed Inuit and started assembling our new CSS library on a static front-end by building out every component on Sundance.org. In order for our team to be able to review and contribute, I shared the repo and hosted it as a live web page on Github Pages (repo, live version). Since we were building this static set of front-end components anyway, it made perfect sense for us to turn this into our living style guide. Complete with design documentation, live examples and code snippets, this is now a great resource for our developers for future projects.


And we were off to the races!

At the time that I am writing this, we're putting the finishing touches are some of the primary elements before moving forward with integrating it into the production site. I'll be sure to update this post with anything conclusive. So far, so good!

Epilogue

We're recognizing more and more that having developers present for design is critical to a project's success. For similar reasons, having designers who can help manage front-end code adds a lot of utility and is empowering for teams. Rather than trying to solve the "handoff" problem, the "handoff" itself almost goes away entirely.

Teams are becoming more and more cross-functional. As a result, I think the gap between designers and developers will close even further and breed better communication, more empathy, and ultimately better work.
